# Obsidian Markdown Reviewer

## 목적

이 스킬은 Obsidian으로 작성한 마크다운(.md) 문서를 **블로그 글처럼 읽기 좋은
형태**로 정리하고, **git push 전에 변경된 문서들을 한 번에 점검**하는 데 목적이
있다.  
Claude는 문장 표현, 문단 구조, 프로그래밍 개념 및 코드 사용 방식까지 함께
검토하고, 먼저 **PLAN 모드로 변경 계획**을 제시한 뒤, 사용자가 승인한 부분만
반영한 개선본을 생성한다.

---

## 기본 원칙

- 문체: 문서는 **평어체**로 작성한다.
  - 예: "~한다", "~이다", "~하자" 등의 형태를 사용한다.
- 형식: 노트는 **블로그 글**처럼 줄글 위주로 구성하고, 필요할 때만 소제목을
  사용한다.
- 구조: 기승전결이 분명해야 하고, 글을 통해 **전달하고자 하는 메시지가
  명확**해야 한다.
- 정확성:
  - 프로그래밍 개념 설명이 사실과 다르거나 모호한 부분을 찾아 바로잡는다.
  - 프로그래밍 언어의 사용 방식이 잘못된 코드 예시는 수정하거나 주석으로
    바로잡는다.
- 존중: 사용자의 원래 의도와 톤을 최대한 유지하면서, 가독성과 정확성을 높이는
  방향으로 수정한다.

---

## 명령어 프로토콜

이 스킬은 아래 문자열 패턴을 **명령어로 인식**한다.  
가능하면 자연어 대신 이 명령어를 사용하도록 한다.

### 1. obs-review: PLAN

- 목적: git push 전에 **변경된 .md 파일 목록을 기준으로 개선 계획만** 먼저 받기
  위한 명령어이다.
- 사용 예:
  - `obs-review: PLAN`
- 동작:
  1. 사용자는 터미널에서 `bash scripts/list_changed_md.sh`를 실행해,
     변경된 .md 파일 목록을 얻는다.
  2. 그 출력 결과를 Claude에게 붙여넣는다.
  3. Claude는 각 파일에 대해:
     - 글의 주제와 핵심 메시지
     - 예상되는 문제점(문장, 구조, 개념, 코드)
     - 파일별 개선 계획(어느 부분을 어떻게 바꿀지)
     를 정리해 PLAN을 제시한다.
  4. Claude는 실제 텍스트를 수정하지 않고, 계획만 설명한다.
  5. 사용자는 "이 계획대로 진행해도 된다/안 된다", "이 파일은 제외하자" 등을
     답한다.

### 2. obs-review: APPLY

- 목적: 직전에 합의한 PLAN을 바탕으로, 실제 **개선된 마크다운 텍스트**를
  생성하기 위한 명령어이다.
- 사용 예:
  - `obs-review: APPLY`
- 동작:
  1. Claude는 마지막 PLAN에서 사용자가 승인한 파일과 수정 범위를 기준으로,
     각 파일의 개선본을 생성한다.
  2. 각 파일에 대해:
     - 간단한 변경 요약
     - 최종 개선된 마크다운 전체
     를 제공한다.
  3. 사용자는 개선본을 수동으로 파일에 반영하거나, 별도 스크립트를 이용해
     적용한다.

### 3. obs-review: REVIEW file=경로/파일명.md

- 목적: 특정 단일 파일만 검토하고 개선하고 싶을 때 사용한다.
- 사용 예:
  - `obs-review: REVIEW file=notes/async-await.md`
- 동작:
  1. 사용자는 `bash scripts/show_file.sh notes/async-await.md`를 실행해 해당 파일
     내용을 Claude에게 붙여넣는다.
  2. Claude는 아래 절차로 진행한다.
     - 현재 글의 요약
     - 문제 진단 (문장/구조/개념/코드 관점)
     - PLAN 제시(어디를 어떻게 바꿀지)
     - 사용자의 승인 여부 확인
     - 승인된 범위에 대해 개선본 생성

## scripts 디렉터리 활용 가이드

이 스킬은 다음 스크립트들이 존재한다고 가정하면 훨씬 효율적으로 동작한다.

### scripts/list_changed_md.sh

- 역할: 현재 git 저장소에서 **변경된 .md 파일 목록**을 한 줄에 하나씩 출력한다.
- 추천 동작:
  - staged 파일 (`--cached`) 기준이거나, working tree 기준 등 사용자가 원하는
    기준으로 구현한다.
- 사용 예:
  - `bash scripts/list_changed_md.sh`

### scripts/show_file.sh

- 역할: 인자로 받은 파일 경로의 내용을 표준 출력으로 보여준다.
- 사용 예:
  - `bash scripts/show_file.sh notes/async-await.md`

Claude는 PLAN/REVIEW 과정에서 사용자에게 다음과 같이 안내할 수 있다.

- "먼저 `bash scripts/list_changed_md.sh`를 실행해, 결과를 나에게 붙여넣어 달라."
- "특정 파일 내용을 보고 싶다면, `bash scripts/show_file.sh 경로/파일명.md` 실행
  결과를 붙여넣어 달라."

## 단일 문서 정리 플로우

명령어: `obs-review: REVIEW file=경로/파일명.md`

1. 사용자가 해당 파일 내용을 Claude에게 제공한다.
2. Claude는 글을 읽고, 다음을 수행한다.
   - 글의 주제와 전달하려는 메시지를 한 단락으로 요약한다.
   - 기승전결 구조를 점검한다.
   - 어색한 문장, 비문, 반복 표현, 논리 연결이 약한 부분을 찾는다.
   - 프로그래밍 개념 설명과 코드 사용 방식이 올바른지 확인한다.
3. Claude는 **PLAN**을 제시한다.
   - "도입부를 이렇게 정리하겠다."
   - "중간에 빠진 개념 X를 보충 설명으로 추가하겠다."
   - "코드 예시 Y를 더 현대적인 사용법으로 바꾸겠다."
   - "결론에서 배운 점을 명확히 정리하겠다."
4. Claude는 사용자에게 다음을 확인한다.
   - "위 계획대로 문서를 개선해도 되는지?"
   - "손대지 않았으면 하는 부분이 있는지?"
5. 사용자가 승인한 경우, Claude는 전체 문서를 평어체/블로그 형식으로 재작성한다.
   - 줄글 기반
   - 불필요한 헤더/리스트 최소화
   - 기승전결과 메시지가 분명한 구조

## git push 전 변경 파일 전체 검토 플로우

명령어: `obs-review: PLAN` → (검토 후) `obs-review: APPLY`

1. 사용자는 터미널에서 `bash scripts/list_changed_md.sh`를 실행해, 변경된
   .md 파일 목록을 Claude에게 붙여넣는다.
2. Claude는 각 파일에 대해:
   - 파일 경로 / 제목
   - 글의 주제와 핵심 메시지(가능하면)
   - 예상되는 문제점과 개선 포인트
   를 정리해 **PLAN**을 만든다.
3. Claude는 PLAN만 제시하고, 실제 수정은 하지 않는다.
4. 사용자는:
   - "이 파일은 이번엔 건너뛰자"
   - "이 파일은 문장만 다듬고, 개념은 그대로 두자"
   같은 식으로 파일별/범위별로 승인 또는 제외를 결정한다.
5. 사용자가 `obs-review: APPLY`를 입력하면, Claude는 승인된 범위에 대해서만
   개선된 마크다운 텍스트를 생성한다.
6. 사용자는 개선본을 수동으로 파일에 반영하거나, 별도 도구를 이용해 반영한다.

## 문장 및 구조 개선 기준

Claude는 다음 기준으로 문서의 질을 개선한다.

- 어색한 문장, 번역투, 중복 표현을 자연스럽고 간결한 한국어로 정리한다.
- "한다 / 이다 / 하자" 어투를 유지하면서, 문서 전체에서 일관되게 유지한다.
- 한 문단에서 다루는 주제는 하나로 유지하고, 주제가 바뀌면 문단을 분리한다.
- 블로그 글처럼, 독자가 처음부터 끝까지 읽을 수 있도록 흐름을 정리한다.
- 도입부에서:
  - 왜 이 글을 쓰는지
  - 어떤 문제를 다루는지
  를 분명히 밝힌다.
- 본문에서:
  - 핵심 개념, 시도한 접근, 코드 예시를 순서대로 정리한다.
- 정리 파트에서:
  - 이번 글에서 얻은 인사이트나 교훈을 한두 문단으로 요약한다.
  - 참고한 문헌이 있으면 반드시 아래에 출처 링크를 명시한다.

## 프로그래밍 개념 및 코드 검토 기준

- 개념이 잘못되었거나 애매하게 설명된 경우:
  - 문제가 되는 부분을 짧게 지적하고
  - 더 정확한 설명을 평어체로 제시한다.
- 코드 예시가 문법적으로 잘못되었거나, 실제로는 잘 쓰이지 않는 방식이면:
  - 수정된 코드 예시를 함께 제공한다.
  - "이 예시는 개념 설명용이라 간단히 작성한다"는 식의 전제도 붙여준다.
- 여러 언어/프레임워크가 섞여 있을 경우:
  - 각 언어/프레임워크 맥락에서 설명을 분리하거나, 주석으로 명시한다.

---

## 트러블슈팅

- 문서가 너무 길어지면:
  - 반복 설명을 통합하고, 핵심 예시 위주로 재구성한다.
- 너무 공격적으로 내용을 바꾸게 느껴지면:
  - 큰 구조 변경 전에는 반드시 PLAN 단계에서 사용자 동의를 구한다.
  - 원래의 톤과 관점을 유지한 채, 표현과 구조만 정리하는 쪽을 우선한다.
